# 联接基金自动穿透功能说明

## 功能概述

在 `app/services/fund_service.py` 的 `get_fund_portfolio()` 函数中，新增了**联接基金自动穿透**功能。当检测到基金的第一大重仓是 ETF 且权重超过 60% 时，系统会自动递归查询该 ETF 的底层持仓，返回真实的股票列表而非 ETF 本身。

## 核心逻辑

### 1. ETF 识别算法

通过 `_is_etf_code(code, name)` 函数判断是否为 ETF：

```python
def _is_etf_code(code: str, name: str = "") -> bool:
    # 方法1: 代码规则
    # - 上交所 ETF: 51xxxx, 56xxxx
    # - 深交所 ETF: 15xxxx, 58xxxx
    if len(code) == 6 and code[:2] in ['51', '15', '56', '58']:
        return True
    
    # 方法2: 名称包含 "ETF" 字样
    if 'ETF' in name.upper():
        return True
    
    return False
```

### 2. 穿透触发条件

在获取基金持仓后，自动检查：

1. **第一大重仓识别**：按权重排序，取权重最高的持仓
2. **ETF 判定**：判断该持仓是否为 ETF（代码或名称符合规则）
3. **权重阈值**：权重 > 60%（认为是联接基金特征）
4. **递归深度**：当前深度 < 3（防止无限递归）

### 3. 穿透执行流程

```
查询基金 005963
    ↓
获取持仓列表
    ↓
检查第一大重仓
    ↓
发现: 半导体50ETF (512480) 权重 95.80%
    ↓
判定: 是 ETF 且权重 > 60%
    ↓
🔍 自动穿透: 递归查询 512480 的持仓
    ↓
✅ 返回: ETF 底层股票（寒武纪、中芯国际等）
```

## 使用示例

### 基本用法

```python
from app.services.fund_service import get_fund_portfolio

# 查询联接基金，自动穿透

for holding in result["holdings"]:

```

## 防护机制

### 1. 递归深度限制

```python
MAX_DEPTH = 3  # 最大递归深度
```

防止无限递归或过深的嵌套（联接基金 → ETF1 → ETF2 → ...）

### 2. 失败降级

如果穿透失败（ETF 持仓为空或出错），自动回退到原始持仓数据：

```python
if etf_portfolio.get("holdings"):
    return etf_portfolio  # 使用穿透后的数据
else:
    print(f"⚠️ ETF {code} 持仓为空，使用原始数据")
    return original_data  # 回退到原始数据
```

### 3. 错误处理

所有 API 调用都包裹在 try-except 中，确保单个查询失败不影响整体流程。

## 适用场景

✅ **联接基金**：第一大重仓是 ETF，权重 > 60%  
✅ **FOF 基金**：持有大量某只 ETF  
✅ **指数增强基金**：主要跟踪某个 ETF  

❌ **不适用**：  
- 第一大重仓不是 ETF  
- ETF 权重 ≤ 60%（分散持仓）  
- 已经是底层股票构成的基金

## 技术细节

### ETF 代码规则

| 交易所 | 代码前缀 | 示例 |
|--------|----------|------|
| 上交所 | 51xxxx   | 512480 (半导体50) |
| 上交所 | 56xxxx   | 560050 (科创50) |
| 深交所 | 15xxxx   | 159995 (芯片ETF) |
| 深交所 | 58xxxx   | 588000 (科创板) |

### 权重阈值说明

- **> 60%**：认为是纯联接基金，自动穿透
- **30% ~ 60%**：可能是混合持仓，不穿透（避免误判）
- **< 30%**：分散持仓，不穿透

可根据实际情况调整阈值：

```python
# 修改阈值（不建议低于 50%）
if _is_etf_code(top_code, top_name) and top_weight > 50:  # 改为50%
    # 执行穿透...
```

## 性能影响

- **额外请求**：最多增加 1 次 API 调用（查询 ETF 持仓）
- **响应时间**：增加约 0.5 ~ 1 秒（取决于网络）
- **成功率**：如果 ETF 数据缺失，自动降级到原始数据

## 注意事项

⚠️ **数据时效性**：ETF 持仓也是季度更新，可能与实时行情有延迟  
⚠️ **跨市场**：港股 ETF 代码规则可能不同，需要依赖名称判断  
⚠️ **特殊基金**：LOF、分级基金等复杂结构未覆盖
